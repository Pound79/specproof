# 段階リリース: 純バグのパッチ → v2 設計変更 → dogfood で実証

ここまでの設計判断（ADR 0001–0006）は現行プラグインに対しほぼ全て破壊的変更を含む。一方
本ツールは実在の本番 E2E スイートで既に dogfood 済みであり、無計画な一括リリースは既存の
採用リポを壊しかねない。また監査の I1–I3 / I5 / I6 / I8 は設計と無関係の**純バグ**で、思想の
議論抜きで即修正でき、全採用者を即解放できる。

そこで 3 段階に分ける:

- **Phase A（パッチ・即時・破壊ゼロ・設計判断なし）**: 純バグのみ。I1（公開 + CI publish、当面は
  具体的 git-dep 手順を README に）・I2（playwright テンプレに traceability コマンド）・I3（Node 24+ 統一）・
  I5/I6（契約の誤マッピング・case 統一）・I8（driftLinkCount）・I10（code-fence）・I20（flutter README）。
- **Phase B（v2・設計変更）**: ADR 0001–0006 の実装。破壊的 enforcement は
  「同語反復ゲート」「空リンク」の 2 つに限定し、他は加算的 / soft 起点。
- **Phase C（実証）**: 実在の本番 E2E スイートで v2 を dogfood し、**既存シナリオが green のまま
  通ること**を受け入れ基準にする。

## Considered Options

- **全部まとめて一括の大改修 v2** — 却下。純バグの解放が設計議論に人質を取られ、既存採用リポの
  受け入れが遅れる。本番への破壊リスクも一括では大きい。

## Consequences

- 純バグの即時解放で新規採用を今すぐ受け入れ可能にする。
- 設計変更は dogfood 実証とセットで安全に出せる。
- 改善の実行順が「バグ → 設計 → 実証」に確定する（ロードマップは `docs/roadmap.md`）。
