# spec が無いリポでは impl↔feature の 2 点トレーサビリティを正規モードとする

bdd-kit の中核は spec↔impl↔feature の 3 点トレーサビリティだが、対象リポにリポ内 markdown spec
（`##` 見出しセクション）が無いケースは多い（specが Notion / Figma / 人の頭の中にある等）。
コードから権威ある spec を自動生成する（impl→spec）と、impl→feature 再生成と同型の同語反復
（コードの誤りを検知できない仕様）を生むため**禁止**する。

代わりに intake では次の順でフォールバックする:
- **(b) spec 骨子抽出（デフォルト）**: AI は「見出し＋観測可能トピックの骨子」までを impl から起こし、
  中身の権威（なぜ・正しさ）は人間が確定してから spec→feature。`impl→骨子→人間が著した spec` であり
  `impl→spec` ではない（人間が権威になることで同語反復を断ち切る）。
- **(a) 2 点運用に正式降格（フォールバック）**: spec を用意できないなら impl↔feature の 2 点
  トレーサビリティとして正規に動かす。drift は spec 側を「未アンカー（spec 側 drift は未ガード）」と
  **明示報告**する。

## Considered Options

- **impl→spec 自動生成** — 却下。impl→feature と同型の同語反復。
- **spec の無いリポは対象外** — 却下。他チーム展開のほとんどの入口で詰む。

## Consequences

- マニフェストは spec 欠落リンクを「欠陥（missing）」ではなく「正規の 2 点状態」として許容する必要がある。
  現 methodology §4 ドリフト決定表は missing を欠陥扱いするため、ここは要改修。
- 「3 点トリプル」は現実には「2 点ペア」へ縮退しうる。これを劣化でなく正規モードとして
  価値提案でも正直に語る。
