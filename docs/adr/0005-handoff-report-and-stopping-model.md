# オーケストレータの出力契約（ハンドオフレポート）と可逆/不可逆で分ける停止モデル

「導入して」一声の価値は、最後に人間へ渡す提示物で決まる。特に他チーム展開では、ここが曖昧だと
「結局何をすればいいか」が伝わらない。

## ハンドオフレポート（固定フォーマット・brownfield/greenfield 共通）

オーケストレータは散文ではなく固定セクションのレポートを常に出す:

1. **やったこと** — scaffold・作成ドラフト・追加リンク。**provenance を明記**（どれがドラフトで
   どれが blessed 済みか）。
2. **到達した検証** — generate / typecheck / lint / smoke を **PASS / FAIL / SKIPPED(理由)** で。
   **正直性契約**: env が無く smoke 未実行なら green と偽らない（監査 I21 の穴を塞ぐ）。
3. **決めてほしいこと（裁定）** — 振る舞い矛盾の権威側（spec↔impl どちらが正）・`@skip` の sign-off・
   adapter/mode 確認。各々を具体的な問いとして。
4. **やってほしいこと（作業）** — `.env`/認証セットアップ・install・spec の WHY 加筆・`@fixme` の自動化・
   **testability backlog**（作るべき test seam）。
5. **完了ダッシュボード** — ドメイン毎に green / `@fixme`(→0) / `@skip`(sign-off 済?)＝ ADR 0002 の done 指標。
6. **次の一手** — 次に打つ単一コマンド。

「決めてほしいこと（裁定）」と「やってほしいこと（作業）」の分離が肝。北極星の 2 語に正確に対応する。

## 停止モデル（可逆/不可逆で分ける）

strength #5（scratch/features 分離）に対応させる:

- **可逆な作業**（scaffold・ドラフト生成・矛盾検出・リンク stub）は**全ドメイン一括**で進め、
  **統合ハンドオフレポート 1 枚**を提示する（「一声で進む」体感）。
- **不可逆な作業**（bless = featuresDir への移動・implement）は**ドメイン毎の明示ゲート**。
  査読なしに権威を刻ませない。

## Consequences

- 多ドメイン brownfield でも「最初に一括で全部見える → 重要な所だけ人間が 1 つずつ承認」になり、
  対話の slog と rubber-stamp の両方を避けられる。
- レポート②の正直性契約は ADR 0004 の検証層と一致させる（SKIPPED は嘘の green を作らない）。
